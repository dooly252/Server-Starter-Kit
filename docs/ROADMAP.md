# LinkHub 개발 로드맵

## 📋 전체 개발 순서

```
1단계: 프로젝트 골격 (구조, 환경 설정)
   ↓
2단계: 공통 모듈 (모든 기능에서 쓰는 것들)
   ↓
3단계: 핵심 기능 (가장 중요한 기능)
   ↓
4단계: 추가 기능 (부가적인 기능)
```

## 1단계: 프로젝트 골격 (구조, 환경 설정)

### 📌 목표
Next.js 프로젝트의 기본 틀을 완성하고 필수 환경 설정을 마치기

### ✅ 작업 목록

1. **Next.js 15 프로젝트 초기화**
   - `create-next-app` 으로 새 프로젝트 생성
   - TypeScript, Tailwind CSS 설정

2. **의존성 패키지 설치**
   - shadcn/ui 초기화
   - Lucide React 설치
   - Notion SDK 설치

3. **환경 변수 설정**
   - `.env.local` 파일 생성
   - Notion API Key, Database ID 설정

4. **디렉토리 구조 구성**
   ```
   src/
   ├── app/                 # 페이지 및 레이아웃
   ├── components/          # 공통 컴포넌트
   ├── lib/                 # 유틸리티 함수
   ├── services/            # API 서비스 (Notion 연동)
   └── types/               # TypeScript 타입 정의
   ```

5. **Git 브랜치 전략 설정**
   - main (프로덕션), develop (개발) 브랜치 생성

### 🤔 왜 이 순서인가?

**필수 전제 조건**
- 모든 개발 작업의 기초 (없으면 다음 단계 불가능)
- 환경 설정이 없으면 Notion API 연동 불가
- 디렉토리 구조가 없으면 파일 조직화 불가능

**개발 효율성**
- 초반에 올바른 구조를 잡으면 이후 개발 속도 향상
- 나중에 구조를 바꾸는 것보다 처음부터 올바르게 설정
- 일관된 구조는 버그 감소 및 유지보수 용이

### ⏱️ 예상 소요 시간
- **3~4시간**

### ✔️ 완료 기준
- [ ] Next.js 프로젝트 생성 완료
- [ ] 필수 패키지 설치 완료
- [ ] `.env.local` 파일 생성 및 설정 완료
- [ ] 디렉토리 구조 생성 완료
- [ ] 기본 레이아웃 페이지 작동 확인

## 2단계: 공통 모듈 (모든 기능에서 쓰는 것들)

### 📌 목표
전체 프로젝트에서 공통으로 사용할 모듈과 컴포넌트 개발

### ✅ 작업 목록

1. **Notion API 서비스 모듈 개발**
   - Notion Client 초기화
   - 데이터베이스에서 링크 조회 함수
   - 데이터 캐싱 로직
   - 타입 정의 (Link, Category 등)

2. **공통 UI 컴포넌트**
   - Header 컴포넌트 (네비게이션, 로고)
   - Footer 컴포넌트
   - Layout 컴포넌트 (Header + Footer 래퍼)
   - Loading, Error 상태 컴포넌트

3. **유틸리티 함수**
   - 날짜 포맷팅
   - URL 유효성 검증
   - 텍스트 잘라내기 (truncate)

4. **타입스크립트 타입 정의**
   ```typescript
   type Link {
     id: string;
     title: string;
     url: string;
     category: string;
     description: string;
     tags: string[];
     createdAt: Date;
     pinned: boolean;
   }
   ```

### 🤔 왜 이 순서인가?

**모든 기능의 기반이 됨**
- Notion API 모듈 없이는 어떤 기능도 데이터를 받아올 수 없음
- 공통 컴포넌트(Header, Footer, Layout)는 모든 페이지에서 필요
- 타입 정의가 없으면 3단계 개발 시 계속 정의해야 함

**개발 속도 및 품질**
- 공통 모듈을 먼저 만들면 개별 기능 개발 시 재사용 (중복 코드 제거)
- 일관된 타입 정의로 버그 감소 및 유지보수 용이
- API 서비스 계층을 분리하면 나중에 변경이 용이

### ⏱️ 예상 소요 시간
- **4~5시간**

### ✔️ 완료 기준
- [ ] Notion API 서비스 모듈 완성
- [ ] Notion에서 링크 데이터 조회 성공
- [ ] 공통 UI 컴포넌트 3개 이상 완성
- [ ] 타입 정의 완료
- [ ] 데이터 캐싱 로직 작동 확인

## 3단계: 핵심 기능 (가장 중요한 기능)

### 📌 목표
LinkHub의 가장 중요한 기능인 링크 조회, 필터링, 검색 완성

### ✅ 작업 목록

1. **홈 페이지 (/)** - 모든 링크 표시
   - Notion에서 링크 데이터 조회
   - 카테고리별 그룹화
   - 링크 카드 렌더링
   - 반응형 그리드 레이아웃

2. **링크 카드 컴포넌트**
   - 제목, 설명, 카테고리, 태그 표시
   - 링크 클릭 시 새탭 오픈
   - 아이콘 (Lucide React)으로 카테고리 표시
   - 호버 이펙트

3. **카테고리 필터링**
   - 상단 필터 버튼 (All, 개발, 디자인, 뉴스, 도구)
   - 필터 클릭 시 해당 카테고리만 표시
   - 활성 필터 상태 표시

4. **검색 기능**
   - 검색 입력창
   - 키워드로 링크 제목/설명 검색
   - 실시간 검색 결과 업데이트

5. **모바일 반응형 디자인**
   - 모바일: 1열 레이아웃
   - 태블릿: 2열 레이아웃
   - 데스크톱: 3~4열 레이아웃

### 🤔 왜 이 순서인가?

**의존성 해결**
- 공통 모듈(Notion API, 공통 컴포넌트) 없이는 구현 불가능
- 2단계가 완료되어야 데이터를 받아올 수 있음

**비즈니스 가치**
- 사용자가 가장 자주 사용하고 기대하는 기능
- MVP(최소 기능 제품) 정의의 중심
- 이 기능이 완성되어야 프로젝트의 가치를 증명할 수 있음

**개발 우선순위**
- 핵심 기능 완성 후에만 추가 기능의 필요성 판단 가능
- 사용자 피드백 수집 후 추가 기능 결정

### ⏱️ 예상 소요 시간
- **5~6시간**

### ✔️ 완료 기준
- [ ] 홈 페이지에서 Notion 데이터 표시 완료
- [ ] 링크 카드 UI 완성 (모든 필드 표시)
- [ ] 카테고리 필터링 기능 작동
- [ ] 검색 기능 작동
- [ ] 모바일/태블릿/데스크톱 반응형 확인
- [ ] 링크 클릭 시 새탭 오픈 확인

## 4단계: 추가 기능 (부가적인 기능)

### 📌 목표
사용자 경험을 향상시키는 부가 기능 구현 (선택 사항)

### ✅ 작업 목록

1. **상세 페이지 (Optional)**
   - `/links/[id]` 동적 라우트
   - 링크 상세 정보 표시
   - 관련 링크 추천

2. **태그 기반 필터링**
   - 모든 태그 목록 표시
   - 태그 클릭 시 필터링
   - 다중 태그 선택 가능

3. **정렬 기능**
   - 최신순, 오래된순
   - 알파벳순
   - 고정된 항목 상단 표시

4. **다크모드**
   - Next.js의 next-themes 활용
   - 라이트/다크 토글

5. **즐겨찾기 기능** (선택)
   - LocalStorage에 저장
   - 즐겨찾기 페이지

### 🤔 왜 이 순서인가?

**핵심 기능 후의 선택 사항**
- 핵심 기능이 완성되어야 추가 기능의 필요성 판단 가능
- MVP 완성 후 사용자 피드백을 바탕으로 우선순위 결정
- 일정 변수에 대응하기 좋은 위치 (추가 기능부터 제외 가능)

**유연한 개발**
- 각 기능이 독립적이므로 추가/제거/순서 변경 용이
- 시간이 부족하면 핵심 기능만 우선 배포 가능
- 사용자 반응에 따라 선택적으로 개발

### ⏱️ 예상 소요 시간
- **3~4시간**

### ✔️ 완료 기준
- [ ] 1개 이상의 추가 기능 완성
- [ ] UI 일관성 유지
- [ ] 모바일 반응형 확인

## 5단계: 최적화 및 배포

### 📌 목표
성능 최적화 및 프로덕션 배포 준비

### ✅ 작업 목록

1. **성능 최적화**
   - 이미지 최적화 (next/image 사용)
   - 번들 크기 분석 및 최적화
   - 캐싱 전략 개선
   - 페이지 로딩 속도 측정

2. **SEO 최적화**
   - Meta 태그 설정
   - Open Graph 태그
   - Sitemap 생성

3. **보안**
   - 환경 변수 보안 검토
   - CORS 설정
   - XSS/CSRF 방지

4. **테스트**
   - 단위 테스트 (주요 함수)
   - E2E 테스트 (주요 시나리오)

5. **배포**
   - Vercel 배포
   - 환경 변수 설정
   - 모니터링 설정

6. **문서화**
   - API 문서 업데이트
   - 개발 가이드 작성
   - 배포 가이드 작성

### 🤔 왜 이 순서인가?

**마지막 단계의 의미**
- 모든 기능 개발이 완료되어야 최적화 대상을 파악할 수 있음
- 기능 개발 중에 최적화를 하면 불필요한 작업이 될 수 있음 (Premature Optimization)

**프로덕션 준비**
- 성능, 보안, 배포는 완성된 제품에서만 의미 있음
- 부분적인 최적화는 전체 영향을 파악할 수 없음

**품질 보증**
- 모든 기능이 정상 작동할 때 통합 테스트 실행
- 배포 전 마지막 검수 단계

### ⏱️ 예상 소요 시간
- **4~5시간**

### ✔️ 완료 기준
- [ ] Lighthouse 점수 90점 이상
- [ ] 모든 테스트 통과
- [ ] Vercel에 배포 완료
- [ ] 프로덕션 URL에서 정상 작동
- [ ] 문서화 완료
